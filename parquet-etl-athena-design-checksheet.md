# Athena 方式 Parquet 化 詳細設計確認シート

## 1. 文書の目的
- 本文書は、[parquet-etl-athena-requirements.md](parquet-etl-athena-requirements.md) と [parquet-etl-athena-design.md](parquet-etl-athena-design.md) の間にある**詳細設計の未確定事項**を、1 項目ずつ確認・確定するための確認シートである。
- 実装前に、判断基準・確認内容・推奨値を明示し、設計レビューで迷わない状態を作ることを目的とする。

## 2. 使い方
- 各項目について、まず**推奨値**を確認する。

- 問題なければ**決定欄**を更新し、設計書へ反映する。
- 別案を採用する場合は、理由と影響範囲をメモに残す。

## 4. 項目別確認シート

### 4.1 Parquet テーブルの列型
- 目的:
  - Parquet テーブルで扱う列型を確定し、SQL の書きやすさと変換失敗リスクのバランスを取る。
- 推奨値:
  - `log_date`: `string`
  - `log_time`: `string`
  - `srcip`: `string`
  - `dstip`: `string`
  - `srcport`: `int`
  - `dstport`: `int`
  - `proto`: `int`
  - `action_raw`: `string`
  - `policyid`: `int`
  - `year`: `string`
  - `month`: `string`
  - `day`: `string`
- 推奨理由:
  - 既存 raw テーブルとの整合を保ちつつ、port / protocol / policy ID は数値比較しやすくなるため。
  - `log_date` / `log_time` を文字列にしておくと、初期段階では型変換エラーを避けやすい。
- 別案:
  - 全列 `string`
  - リスク:
    - 型変換失敗は減るが、数値比較や集計がしにくい。
  - 運用コスト:
    - SQL ごとに `cast` が必要になりやすい。
  - 学習難易度:
    - 低い。
- 判断基準:
  - 実ログに欠損や異常値がどの程度あるか。
  - 数値比較や集計をどこまで重視するか。
- 確認内容:
  - raw テーブル上で `srcport`, `dstport`, `proto`, `policyid` に異常値や空文字がないか。
  - 今後 `policyid` や `proto` で集計したい要件があるか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - `year/month/day` は raw テーブルと同じ `string` を維持する。
  - 理由は、ゼロ埋め表現と raw / Parquet の SQL 記述を揃え、運用と比較検証を単純にするため。

### 4.2 Parquet の圧縮方式
- 目的:
  - Parquet 出力時の圧縮方式を確定し、検索速度とストレージ効率のバランスを取る。
- 推奨値:
  - `SNAPPY`
- 推奨理由:
  - Athena / Parquet では一般的で、検索性能と圧縮率のバランスが良いため。
- 別案:
  - `GZIP`
  - リスク:
    - 圧縮率は高いが、検索速度が落ちやすい。
  - 運用コスト:
    - 低い。
  - サービス費用:
    - ストレージはやや有利。
  - 学習難易度:
    - 低い。
- 判断基準:
  - 検索速度を優先するか、ストレージ削減を優先するか。
- 確認内容:
  - 本対応では検索速度向上が必須要件であることを再確認する。
  - raw と Parquet の保存コスト差をどこまで重視するか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - 検索速度向上が必須要件であるため、Parquet の圧縮方式は `SNAPPY` を採用する。
  - `GZIP` はサイズ面では有利だが、今回の優先事項は検索性能のため採用しない。

### 4.3 ETL 用 Athena WorkGroup 分離
- 目的:
  - 検索用クエリと ETL 用クエリの責務・権限・運用を分けるか確定する。
- 推奨値:
  - ETL 専用 WorkGroup を追加する
  - 例:
    - 検索用: `fw-log-analytics-wg`
    - ETL 用: `fw-log-analytics-etl-wg`
- 推奨理由:
  - 検索と ETL のクエリ履歴、失敗、コストを分けて見やすくなるため。
  - ETL 実行ロールの権限を WorkGroup 単位で絞りやすい。
- 別案:
  - 既存 WorkGroup を共用する
  - リスク:
    - 検索と ETL の実行履歴が混ざる。
  - 運用コスト:
    - 低い。
  - サービス費用:
    - WorkGroup 自体に追加料金はない。
  - 学習難易度:
    - 低い。
- 判断基準:
  - 権限分離と運用の見やすさをどこまで重視するか。
- 確認内容:
  - ETL 実行履歴を検索利用者の履歴と分けて見たいか。
  - ETL 用に結果出力先 prefix を分けたいか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - ETL 用 WorkGroup を検索用 WorkGroup と分離する。
  - 想定名は `fw-log-analytics-etl-wg` とし、検索用の `fw-log-analytics-wg` とは役割を分ける。
  - 理由は、クエリ履歴、失敗確認、コスト把握、権限分離を行いやすくするため。

### 4.4 Lambda の実装言語・実行設定
- 目的:
  - ETL 実行 Lambda の runtime と基本実行設定を確定する。
- 推奨値:
  - 言語: Python
  - timeout: 600 秒
  - memory: 256 MB から開始
  - retry: EventBridge Scheduler の標準再試行を利用し、Lambda 側で明示的に失敗終了する
- 推奨理由:
  - boto3 で Athena / S3 / Glue 連携を書きやすく、学習しやすいため。
  - Lambda は重変換をするのではなく、Athena クエリの起動と監視が中心のため、初期メモリは小さめでよい。
- 別案:
  - Node.js
  - リスク:
    - 実装は可能だが、現時点の学習導線では Python の方が資料が多い。
  - 運用コスト:
    - 同等。
  - 学習難易度:
    - Python よりやや高い。
- 判断基準:
  - 学習しやすさ
  - boto3 / SDK の使いやすさ
  - Lambda 内での処理量
- 確認内容:
  - Athena ETL の実測で、Lambda が 600 秒以内で十分か。
  - Athena 完了待ちは Lambda で待つか、Step Functions などへ分ける必要があるか。
- 決定欄:
  - [ ] 推奨値を採用する
  - [ ] 別案を採用する
  - [x] 保留
- メモ:
  - 一旦 A 案を採用し、Python / 256 MB / 600 秒で仮設定とする。
  - Athena `INSERT INTO` の実測値を確認後、timeout を再調整する。
  - 実測で 600 秒に対して余裕が大きければ短縮し、逆に逼迫する場合は待機方式の見直しも検討する。

### 4.5 Athena SQL の管理方式
- 目的:
  - ETL に使う Athena SQL をどこで管理するか確定する。
- 推奨値:
  - SQL は Lambda コード埋め込みではなく、別ファイル管理とする
  - 例:
    - `sql/parquet/insert_daily.sql`
- 推奨理由:
  - SQL とコードの責務を分けた方がレビューしやすく、差分も追いやすいため。
- 別案:
  - Lambda コードに文字列として埋め込む
  - リスク:
    - SQL 修正とコード修正が混ざる。
  - 運用コスト:
    - 小規模では低い。
  - 学習難易度:
    - 低い。
- 判断基準:
  - SQL を今後増やすか。
  - SQL のレビュー性をどこまで重視するか。
- 確認内容:
  - 日次用、バックフィル用、再生成用で SQL を分ける必要があるか。
  - パラメータ埋め込み方式をどうするか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - ETL 用 SQL は Lambda コードに埋め込まず、別ファイルで管理する。
  - 例として `sql/parquet/insert_daily.sql` を想定する。
  - 理由は、SQL とコードの責務を分け、レビューと差分管理をしやすくするため。

### 4.6 日次実行の時刻と対象日ルール
- 目的:
  - 日次 ETL がどの時刻に、どの日付を対象に実行されるか確定する。
- 推奨値:
  - 対象日: 前日分
  - 実行時刻: raw 投入と Glue パーティション登録完了後
  - 例:
    - 現時点では upstream が anacron のため固定できないので、仮で `08:00 JST` に設定する
  - 補足ルール:
    - 直近 7 日を確認し、raw があり Parquet 未作成の日を順に catch-up 処理する
    - raw 未着は失敗ではなく skip とする
    - raw があり ETL に失敗した場合のみ fail とする
- 推奨理由:
  - 当日分を扱うと、まだ投入途中のデータを変換するリスクがあるため。
- 別案:
  - 当日分を複数回増分変換する
  - リスク:
    - 運用と整合性が複雑になる。
  - 運用コスト:
    - 高い。
  - サービス費用:
    - クエリ回数が増えやすい。
  - 学習難易度:
    - 高い。
- 判断基準:
  - raw 投入の完了時刻
  - 遅延到着ログの頻度
- 確認内容:
  - 現在の raw 投入が、実測で何時ごろまでに終わるか。
  - 前日分確定扱いで運用できるか。
- 決定欄:
  - [ ] 推奨値を採用する
  - [ ] 別案を採用する
  - [x] 保留
- メモ:
  - upstream が anacron のため、raw アップロード完了時刻は現時点で固定できない。
  - 一旦は安全側の仮設定として、対象日は前日分、ETL 実行時刻は `08:00 JST` とする。
  - raw アップロードと Glue パーティション登録の実測完了時刻を確認後、必要に応じて前倒しまたは後ろ倒しする。
  - 遅延到着ログに対応するため、単純な「前日 1 日だけ処理」ではなく、直近 7 日の未処理日を拾う catch-up 方式を採用する。

### 4.7 初回バックフィルの実行方法
- 目的:
  - 初回の 1 年分 Parquet 化を、誰が何で実行するか確定する。
- 推奨値:
  - 管理者が PowerShell から日付単位でスクリプト実行する
  - 例:
    - PowerShell から Lambda を日付ごとに呼び出す
- 推奨理由:
  - 失敗日の切り分けと再開がしやすいため。
  - 日次運用と同じロジックを使い回しやすい。
  - 既存の学習・運用フローが PowerShell ベースで、補助スクリプトをリポジトリ管理しやすいため。
- 別案:
  - CloudShell から日付ループを回す
  - リスク:
    - ローカル Git 管理している補助スクリプトとの距離が出やすい。
  - 運用コスト:
    - 中。
  - サービス費用:
    - 差はほぼない。
  - 学習難易度:
    - PowerShell よりやや高い。
- 判断基準:
  - PoC としての分かりやすさ
  - 失敗時の再開しやすさ
- 確認内容:
  - 初回バックフィルは管理者の手動起動でよいか。
  - バックフィルにどの程度の時間がかかっても許容できるか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - 初回バックフィルは PowerShell 補助スクリプトから Lambda を日付単位で呼び出す方式とする。
  - 理由は、学習しやすく、失敗日の再開と Git 管理がしやすいため。

### 4.8 再生成時の削除範囲と安全策
- 目的:
  - 対象日再生成時の削除範囲と誤削除防止策を確定する。
- 推奨値:
  - 削除範囲は対象日の Parquet prefix のみ
  - 削除前に raw 側対象日の存在確認を行う
  - 実行ログに対象日と削除 prefix を必ず出す
  - 削除前に dry-run 相当の確認ログを出す
- 推奨理由:
  - 再生成の安全性を確保しつつ、重複を防げるため。
- 別案:
  - Parquet 側を削除せず追加のみ
  - リスク:
    - 重複データが発生しやすい。
  - 運用コスト:
    - 低いが、データ整合性が崩れやすい。
  - 学習難易度:
    - 低い。
- 判断基準:
  - 誤削除防止をどこまで強くしたいか
  - 重複排除をどこまで確実にしたいか
- 確認内容:
  - 削除前に dry-run 相当のログを出したいか。
  - raw が存在しない場合は処理を中止する方針でよいか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - 削除対象に誤りがないことを確認しやすくするため、削除前に dry-run 相当のログを必ず出す。
  - raw が存在しない場合は削除・再生成を行わず、中止する。
  - これにより、誤った日付の削除や、原本がない状態での再生成を防ぐ。

### 4.9 性能確認の評価基準
- 目的:
  - Parquet 切替の可否を判断するための性能比較方法を確定する。
- 推奨値:
  - 比較データセットは次の 3 パターンに固定する
    - 1日
    - 30日
    - 365日
  - 比較 SQL は 4 本固定
    - `srcip`
    - `dstip`
    - `action`
    - 期間指定
  - 各 SQL を raw / Parquet それぞれ 3 回実行し、中央値で比較する
  - 比較条件を揃えるため、`ORDER BY` と `LIMIT` は付けない
  - 比較指標:
    - 実行時間
    - スキャンしたデータ量
  - 合格条件:
    - raw よりスキャン量が少ない
    - raw より実行時間が短い
- 推奨理由:
  - ばらつきのある 1 回勝負ではなく、中央値で比較した方が実務的なため。
- 別案:
  - 1 回だけ実行して比較する
  - リスク:
    - Athena の実行ばらつきで誤判定しやすい。
  - 運用コスト:
    - 低い。
  - 学習難易度:
    - 低い。
- 判断基準:
  - 再現性をどこまで重視するか
  - 計測工数をどこまで許容するか
- 確認内容:
  - 比較データセットを何日分で固定するか。
  - `ORDER BY` や `LIMIT` の有無を比較条件で揃えるか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - 比較は 1日 / 30日 / 365日で実施する。
  - 比較条件を揃えるため、`ORDER BY` と `LIMIT` は付けない。
  - 各 SQL は raw / Parquet それぞれ 3 回実行し、中央値で評価する。

### 4.10 標準検索先の切替手順
- 目的:
  - Parquet が性能要件を満たしたあと、利用者向けの標準検索先をどう切り替えるか確定する。
- 推奨値:
  - 性能確認完了後に、次を同一タイミングで更新する
    - `README.md`
    - `runbook/athena-search.md`
    - `runbook/sql-templates.md`
  - raw テーブルはフォールバックとして残す
- 推奨理由:
  - ドキュメントの切替を分散させると、利用者が raw / Parquet のどちらを使うべきか迷いやすいため。
- 別案:
  - README だけ先に更新し、Runbook は後追いにする
  - リスク:
    - 手順不整合が起きやすい。
  - 運用コスト:
    - 中。
  - 学習難易度:
    - 低い。
- 判断基準:
  - 利用者が迷わないこと
  - raw フォールバック手順を残せること
- 確認内容:
  - 切替後も raw テーブルの利用場面を明記するか。
  - Runbook 上で raw をどの章に残すか。
- 決定欄:
  - [x] 推奨値を採用する
  - [ ] 別案を採用する
  - [ ] 保留
- メモ:
  - Parquet が切替条件を満たした後に、README、Runbook、SQL テンプレートを同一タイミングで更新する。
  - raw テーブルはフォールバック用途として残し、標準検索先のみ Parquet 優先へ切り替える。
  - 利用者が raw / Parquet の使い分けで迷わないことを優先する。
